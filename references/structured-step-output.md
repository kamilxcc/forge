# structured-step-output — 结构化编码输出模板

本片段被 implementer Agent prompt 引用。
每完成计划中的一个步骤，按此模板输出一段摘要。

---

## 每步完成后的输出格式

```markdown
---
### ✅ Step N 完成：<步骤标题>

**修改文件**：
- `path/to/file.kt` — <一句话说明改了什么>
- `path/to/another.kt` — <一句话说明改了什么>

**关键决策**：
<如果做了超出计划的设计决策（选了某种实现方案、发现坑并绕开），在此说明 why。
 如果完全按计划执行，写"按计划执行，无额外决策"。>

**注意事项**：
<发现的潜在风险、需要后续关注的点，或需要 reviewer 重点检查的地方。
 无则写"无"。>

**发现但未触碰**（可选，无则删除）：
- `path/to/file.kt:N`：<发现的问题描述>，超出本次任务范围，未处理
<仅在本步骤执行时发现了范围外的问题、坑点、或可优化项时填写。
 不要因为"顺手可改"就填写后再修改——发现即声明，不处理。>

**下一步**：Step N+1 — <下一步标题>
---
```

---

## 所有步骤完成后的汇总格式

```markdown
---
## 🎉 实施完成

**功能**：<Feature 名称>（对应 `features/<slug>.md`）

**变更摘要**：
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `path/to/file.kt` | 新增/修改/删除 | 简短说明 |

**关键决策汇总**：
1. <决策 1>：<why>
2. <决策 2>：<why>

**已知风险 / 建议检查点**：
- <风险 1>
- <风险 2>（无则删除本节）

**未触碰声明**（可选，无则删除）：
<执行过程中发现但未处理的范围外问题，汇总自各步骤的「发现但未触碰」字段。
 格式：
 - `path/to/file.kt:N`：<问题描述>（发现于 Step M）
 这些条目不是偏差，是信息移交——供用户决策是否要补一个任务处理。>

**偏差记录**：
<如果执行过程中发现需要超出计划的修改（发现 bug、设计缺陷、或不可预见的依赖问题），记录在此。
 格式：
 - 文件：`path/to/file.kt`，改动：<描述>，原因：<为什么必须这样改>
 无偏差则删除本节。>

**验收清单**：
| # | 检查项 | 结果 |
|---|--------|------|
| 1 | 所有步骤均已执行 | ✅ / ❌ |
| 2 | 无未声明的计划外改动 | ✅ / ❌ / N/A |
| 3 | 边界约定遵守（plan.md 中"不做什么"） | ✅ / ❌ / N/A |
| 4 | CLAUDE.md 项目规则合规 | ✅ / ❌ / N/A |
| 5 | 范围外发现已显式声明 | ✅ / ❌ / N/A |
| 6 | 现有测试未被覆盖 | ✅ / ❌ / N/A |

**建议下一步**：
- 运行 `/review` 进行代码审查
- 或运行 `/test` 补充测试覆盖
---
```

---

## 使用规则

1. **每完成一个计划步骤**，必须输出一段 Step N 摘要，不可跳过
2. **关键决策必须记录 why**，不能只写 what（这是知识库沉淀的原材料）
3. **不要在步骤摘要里贴完整代码**，只写文件路径 + 一句话说明
4. **所有步骤完成后**，输出一段汇总，并明确提示用户下一步操作
5. **若执行过程中发现计划外改动**，必须在"偏差记录"节中记录，供审查阶段参考
6. **发现范围外问题但不处理时**，在本步骤的「发现但未触碰」字段记录；汇总时合并到「未触碰声明」节——这是信息移交，不是偏差，不要沉默跳过
